Labeling source code changes and comments

First of all, we thank you for your participation in this project.

Your task, if you choose to accept it, is to answer several questions about consistency of changes in Java code with corresponding comment.

Depending on the result of this task, the best tolokers will be permitted to access more complicated tasks of annotating source code and comments.

Labeling example

The examples provided for the annotation will look like this:

Learn more about this example. You may notice that the code changes introduce a new parameter to the method's signature. After the change, the method takes 5 parameters instead of 4. However, the corresponding comment stil have information only about 4 of them.

  • To make the code and comment consistent with each other, we need to add @param with description of newly added parameter to the comment. Therefore, response to the question "Should comment be changed?" is "Yes".

  • The next question is "Must the comment be completely overwritten?" and the answer is "No". The code changes don't make the comment incorrect completely, there is only a need to update it by providing information about a new parameter.

  • The next question is "Should something be changed in @param, @return or @throws part of the comment?". The answer is "Yes", we because we need to add a new @param to the code comment.

  • Finally, we have to choose only Param checkbox, because other options Return and Throws aren't changed in the code.
